Database Profiles

W
Weaviate
Purpose-built Vector DB
● Native
Sparse
BM25 built-in, no extra setup
Dense
HNSW vector index, module-based embeddings
Fusion
RRF + relative score fusion, both native
Strength
Zero infra overhead — one DB does everything. Strong GraphQL API.
Weakness
Less mature BM25 tuning vs Elasticsearch; limited sparse customization
✓ Single Store ✓ RRF Native ✓ Metadata Filter ✓ Multi-tenant
E
Elasticsearch
Search Engine + Vector DB
● Native
Sparse
Industry-leading BM25 / BM25F, full inverted index
Dense
kNN via HNSW (v8+), ELSER sparse neural model
Fusion
RRF native (v8.8+), custom linear combination
Strength
Best-in-class BM25, battle-tested at scale, rich aggregations & analytics
Weakness
Heavy JVM resource usage; expensive at scale; complex ops
✓ Best BM25 ✓ RRF Native ✓ ELSER Neural ~ High Ops Cost
O
OpenSearch
Open-source Search + Vector
● Native
Sparse
BM25 (Lucene-based), same quality as ES
Dense
HNSW + IVF via k-NN plugin
Fusion
Hybrid search query DSL, normalization processors
Strength
Truly open-source (Apache 2.0), AWS-native, Elasticsearch parity
Weakness
Slightly behind ES in ML features; AWS lock-in risk on managed
✓ Apache 2.0 ✓ AWS Native ✓ BM25 Strong ~ ES Feature Lag
Q
Qdrant
Purpose-built Vector DB (Rust)
● Native
Sparse
Sparse vectors (SPLADE / BM42), stored natively
Dense
HNSW, multiple named vectors per point
Fusion
RRF + Distribution-Based Score Fusion (DBSF)
Strength
Blazing fast (Rust), low memory, excellent filtering, SPLADE support
Weakness
Sparse requires you to bring your own SPLADE model; smaller ecosystem
✓ Rust Speed ✓ SPLADE ✓ DBSF Fusion ~ DIY Sparse Model
pg
pgvector
PostgreSQL Extension
◑ Partial
Sparse
PostgreSQL FTS (tsvector/tsquery) — no BM25 by default
Dense
IVFFlat + HNSW via pgvector extension
Fusion
Manual SQL union + custom scoring; RRF via raw SQL only
Strength
Already in your Postgres stack — zero new infra, full SQL power
Weakness
No native BM25, manual fusion logic, poor horizontal scale for vectors
✓ Zero New Infra ✓ Full SQL ✗ No BM25 ✗ Manual Fusion
P
Pinecone
Managed Vector DB (Cloud)
◑ Partial
Sparse
Sparse-dense index via SPLADE — you generate sparse vectors externally
Dense
Proprietary HNSW, fully managed
Fusion
Weighted α blend at query time — no RRF
Strength
Zero ops, serverless scaling, great managed experience
Weakness
Must run SPLADE model yourself; proprietary, costly at high volume
✓ Zero Ops ✓ Serverless ~ SPLADE External ✗ No RRF
R
Redis (RediSearch)
In-memory + Search Module
◑ Partial
Sparse
Full-text search via RediSearch module (BM25-like)
Dense
HNSW vectors via RediSearch VECTOR field
Fusion
Hybrid query via FT.SEARCH — limited fusion control
Strength
Ultra-low latency (in-memory), great if already using Redis stack
Weakness
Memory-bound — expensive for large corpora; fusion is basic
✓ Ultra-low Latency ✓ In-memory ~ Basic Fusion ✗ Memory Cost
M
Milvus
Cloud-native Vector DB
◑ Partial
Sparse
Sparse float vectors (SPLADE / BM25 external) since v2.4
Dense
HNSW, IVF, DiskANN — wide ANN algorithm support
Fusion
Weighted RRF via hybrid search API
Strength
Massive scale (billions of vectors), cloud-native, GPU acceleration
Weakness
Operationally complex (K8s), sparse still maturing, steep learning curve
✓ Billion-scale ✓ GPU Accel ~ Sparse Maturing ~ Complex Ops

Pick by scenario

🏗️ Starting a new RAG project

→
Need hybrid + zero extra infra? Weaviate
→
Want fastest managed cloud start? Pinecone
→
Already have Postgres? pgvector
→
Need high perf + open-source? Qdrant

🏢 Enterprise / existing stack

→
Already run Elasticsearch? Stay — add kNN Elastic
→
AWS shop, open-source needed OpenSearch
→
Redis already in cache layer Redis
→
Postgres shop, small corpus pgvector

📈 Scale & performance focus

→
Billions of vectors, GPU cluster Milvus
→
Massive docs, rich analytics Elastic
→
Sub-5ms p99, memory not a cost Redis
→
Best perf/cost ratio, self-hosted Qdrant
