Hybrid Search · Fusion Strategy

RRF vs Weighted Sum

How to merge dense + sparse results — and which approach wins in what situation

Reciprocal Rank Fusion
RRF(d) = Σ 1 / (k + ranki(d))
k = 60 (smoothing constant)
rank = position in result list
Ignores raw scores entirely. Only the rank position of a document matters. A constant k=60 prevents top-1 docs from dominating everything.
VS
Weighted Sum Fusion
score(d) = α · dense(d)
           + (1−α) · sparse(d)
α ∈ [0, 1] (tunable weight)
Blends the actual normalized scores from each retriever. Requires careful normalization (min-max or softmax) before combining — raw scores are incompatible.

When to use which
✅ Use RRF when…
You have no labeled query data to tune weightsCold-start projects, new domains, prototyping
You're frequently upgrading embedding modelsRRF is immune to score distribution changes
Queries are exploratory / open-endede.g. "explain how transformers work"
You want a safe, robust default with minimal engineeringMost RAG systems start here
Multiple retrievers (3+) need to be combinedRRF handles N sources cleanly; WSF gets messy
✅ Use Weighted Sum when…
You have labeled evaluation data to tune αHuman-annotated query-document relevance pairs
Query types are known and classifiablee.g. keyword vs semantic can be detected at runtime
You need fine-grained control over retriever influencee.g. exact product codes must always prioritize BM25
Domain has stable score distributionsModels rarely change, normalization stays valid
You're optimizing for known-item / lookup queriese.g. "find invoice #INV-2024-0042"

💡

RRF = Safe Default

Start here. No tuning, no normalization headaches, robust across model upgrades. Works great for 80% of RAG use cases.

Weighted Sum = Precision Tool

Use when you have data, stability, and a specific retrieval pattern you want to optimize. Higher ceiling, higher cost to get right.
