🗂️  Indexing Phase Challenges
🔴 High

Dual Index Synchronization

Keeping the dense vector index and sparse BM25 index in sync is hard. A doc added to one but not the other causes inconsistent results.

Atomic writes across 2 separate stores are rarely guaranteed Partial failure → ghost documents or missing entries
🔴 High

Chunking Strategy Affects Both Indexes Differently

Chunk size optimized for semantic embeddings may be too large for precise BM25 keyword matching — one strategy can't serve both equally.

Small chunks: better BM25, but weaker context for embeddings Large chunks: richer semantics, weaker keyword precision May need multi-granularity chunking (expensive)
🟡 Medium

BM25 Vocabulary Drift Over Time

As new documents are indexed monthly, the term-frequency statistics powering BM25 change — old documents get re-scored differently without re-indexing.

Full BM25 re-index needed for accurate IDF recalculation Incremental updates cause score skew between old/new docs
🟡 Medium

Embedding Model Version Lock-in

Upgrading your embedding model requires re-embedding every document from scratch — mixing old/new vectors in one index breaks similarity calculations.

Full re-index cost can be very high at scale Must version-control which model generated which vectors
🟢 Low–Med

Storage & Infra Cost × 2

Hybrid search requires maintaining two indexes simultaneously — storage doubles, and you pay for two query engines if not using a native hybrid DB.

Separate vector DB + Elasticsearch = double operational overhead Native hybrid DBs (Weaviate, Qdrant) reduce but don't eliminate this
⚡  Retrieval Phase Challenges
🔴 High

Score Normalization Before Fusion

Dense scores (cosine similarity: −1 to 1) and sparse scores (BM25: 0 to ∞) live in completely different ranges — you can't combine them without normalization.

Min-max normalization is sensitive to outliers RRF avoids this but discards absolute score information Wrong normalization silently biases one side
🔴 High

Fusion Weight Tuning (α)

Hybrid search blends results as: score = α·dense + (1−α)·sparse. Choosing α is non-trivial and query-type dependent.

Factual/keyword queries → want higher sparse weight Conceptual/semantic queries → want higher dense weight Static α is always a compromise; dynamic α needs ML overhead
🔴 High

Latency: Two Searches in Parallel

Running dense + sparse search simultaneously doubles query complexity. If done sequentially, latency doubles — unacceptable for realtime UX.

Requires parallel execution and async coordination Network round-trips to separate stores compound the issue P99 latency spikes are harder to manage than P50
🟡 Medium

Query Preprocessing Mismatch

The same user query must be processed differently for each retrieval path — embedded for dense, tokenized/stemmed for sparse.

Spelling errors hurt BM25 more than dense embeddings Query expansion helps sparse but can hurt dense precision
🟡 Medium

Re-ranking After Fusion Adds Latency

Fused top-K results often need a cross-encoder re-ranker for final precision — this adds another model inference step in the hot path.

Cross-encoder is slower than bi-encoder retrieval Skipping re-ranking sacrifices quality for speed
⬇ challenges spanning both phases ⬇
🔗

Shared / Cross-Phase Challenges

🧪 Evaluation Complexity

Hard to measure which blend of dense+sparse actually improves recall/precision. Requires labeled query-document datasets per domain.

🌐 Multilingual & Domain Gaps

BM25 is language-aware (tokenization/stemming per language); embedding models vary in multilingual quality. Both need per-language tuning.

🔄 Incremental Update Consistency

When a doc is updated/deleted, both indexes must reflect the change. Stale data in either index causes incorrect retrieval silently.

🏗️ Operational Complexity

Two systems to monitor, scale, back up, and debug. Failure in one degrades results without obvious error — hybrid silent failures are insidious.
